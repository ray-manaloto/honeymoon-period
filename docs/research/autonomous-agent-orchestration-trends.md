# Autonomous-agent orchestration trends

- **Question:** Are durable goal records, leases/single-writer coordination, bounded review-repair-validation loops, checkpoints/event logs, deterministic gates, human/authority interrupts, idempotent retries, and bounded subagent delegation used by current modern agent harnesses and multi-agent orchestration systems? Which are ecosystem trends rather than repository-specific inventions?
- **Status:** Comparative primary-source survey; no implementation, scheduling, configuration, credential, Git, or external-system change is authorized by this report.
- **Accessed / last verified:** 2026-07-19.
- **Symphony snapshot:** upstream `main` commit `7af5a7648c9fbffa08825fe0c0b18be00100aff3`; latest release `v0.0.1` dated 2026-07-18.
- **Research:** linked — extends [Autonomous Codex goal continuation](codex-autonomous-goal-continuation.md) and reuses its [workflow primitives](codex-autonomy-workflow-primitives.md), [quality loops](codex-autonomy-quality-loops.md), and [release/capability inventory](codex-autonomy-release-capability-inventory.md).
- **Last30Days:** linked — root-owned `.build/research/last30days/modern-autonomous-coding-agent-harnesses-and-multi-agent-orchestration-trends-raw-v3.md` plus the focused `.build/research/last30days/openai-symphony-codex-orchestration-release-adoption-reliability-limitations-raw.md`; Symphony community evidence is release-day thin, and discovery is not authority for this survey.
- **Evidence rule:** “native” means an owning source documents the control as a framework/runtime feature; “application-owned” means the source exposes primitives but does not supply the policy; “absent” means not found in the reviewed current owning docs, not proof that an extension cannot add it. Cookbook/tutorial examples are labelled examples, not platform contracts.

## Answer

**Yes, most controls are established practice—but at different layers.** Durable state/checkpoints, replay/resume, retry policies, event histories, bounded execution, human approval, and constrained delegation are now common first-class capabilities in durable workflow engines and leading agent frameworks. OpenAI Symphony is the closest direct coding-orchestrator match: its draft specification combines a repository-owned workflow contract, tracker-driven scheduling, isolated workspaces, one process-local scheduling authority, reconciliation, retries, bounded concurrency, and Codex App Server. LangGraph, Microsoft Agent Framework, and Temporal remain the strongest durable-workflow matches. Codex/Agents SDK, Claude Code, OpenHands, SWE-agent, and CrewAI supply substantial pieces (goals or sessions, traces/trajectories, permissions, retry or turn limits, subagents), but generally do **not** provide a repository-level authority record or distributed single-writer lease.

The proposed repository controller is therefore not a novel collection of ideas. It combines mature workflow-engine controls with coding-agent controls. Its distinctive, locally necessary part is the **policy seam**: a source-controlled, privacy-safe authority record plus a local single-writer lease that governs an agent runtime which does not own the repository’s product, Git, signing, deployment, or relationship-data authority.

## Verified evidence by system

### OpenAI: Codex and Agents SDK

- **Verified:** Codex Goals are persisted *thread* state with an objective, lifecycle and budget/progress accounting. Automatic continuation is bounded by idle/pending-work/user-input conditions, and a no-tool-call continuation suppresses the next automatic continuation. This is an exact bounded-goal mechanism, not durable repository authorization or concurrency control. [Using Goals in Codex](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex)
- **Verified, example not contract:** OpenAI’s iterative-repair Cookbook implements structured review, focused repair, validation, per-attempt records, and termination on success, maximum attempts, unchanged delta, or human review. The report’s proposed quality loop is closely analogous, but the example does not furnish a general Codex service, lease, or completion oracle. [Iterative repair loops](https://developers.openai.com/cookbook/examples/codex/build_iterative_repair_loops_with_codex)
- **Verified:** The Agents SDK supplies sessions, tool schemas/guardrails, handoffs, `max_turns`, tracing, and human approval. Approval creates interruptions; `RunState` can be serialized, approved/rejected, and resumed across runs. Handoffs are model-visible tools and stay within a single run, so they are delegation mechanics—not durable multi-run work arbitration. [Agents SDK overview](https://openai.github.io/openai-agents-python/), [human-in-the-loop](https://openai.github.io/openai-agents-python/human_in_the_loop/), [handoffs](https://openai.github.io/openai-agents-python/handoffs/), [tracing](https://openai.github.io/openai-agents-python/tracing/)
- **Inference:** OpenAI supports the inner agent loop well. The project must own the outer authority record, one-writer behavior, path allowlist, and acceptance contract, as the existing Codex reports concluded.

### OpenAI Symphony

- **Verified:** OpenAI released Symphony `v0.0.1` on 2026-07-18 as an open-source, language-agnostic draft specification plus an experimental Elixir reference implementation. This report reviewed the later same-day `main` snapshot `7af5a7648c9fbffa08825fe0c0b18be00100aff3`. Symphony is explicitly a long-running scheduler/runner that polls an issue tracker, creates one isolated workspace per issue, loads repository-owned policy from `WORKFLOW.md`, launches Codex through App Server, reconciles tracker state, retries failed or stalled work, and exposes structured status. OpenAI reports internal use and a 500% increase in landed pull requests on some teams, but that is a first-party observational claim rather than an independently reproduced benchmark. [OpenAI announcement](https://openai.com/index/open-source-codex-orchestration-symphony/), [reviewed repository snapshot](https://github.com/openai/symphony/tree/7af5a7648c9fbffa08825fe0c0b18be00100aff3), [v0.0.1 release](https://github.com/openai/symphony/releases/tag/v0.0.1)
- **Verified:** The draft spec requires one authoritative in-memory orchestrator state, `claimed`/`running` duplicate-dispatch checks, reconciliation before dispatch, deterministic issue eligibility, global and per-state concurrency limits, `max_turns`, stall/turn timeouts, and exponential-backoff retries. This is a direct precedent for the proposed thin controller, but its one-writer protection is process-local and has neither a TTL nor a worktree/HEAD ownership identity. [Symphony specification](https://github.com/openai/symphony/blob/7af5a7648c9fbffa08825fe0c0b18be00100aff3/SPEC.md)
- **Verified limitations:** Restart recovery is tracker/filesystem-driven; retry timers, running sessions and live worker state do not survive restart, and persisting retry/session metadata is a stated TODO. The Elixir reference keeps blocked approval/operator-input state only in memory. The core spec leaves approval and sandbox posture implementation-defined and does not make external tracker or Git effects idempotent. The README calls Symphony an engineering preview for trusted environments, while the reference implementation is prototype software for evaluation. [specification](https://github.com/openai/symphony/blob/7af5a7648c9fbffa08825fe0c0b18be00100aff3/SPEC.md), [reference implementation](https://github.com/openai/symphony/blob/7af5a7648c9fbffa08825fe0c0b18be00100aff3/elixir/README.md)
- **Verified, example not core contract:** The shipped `WORKFLOW.md` models planning, validation, PR-feedback repair, `Human Review`, `Rework`, and `Merging`. It demonstrates a review-repair-validation policy, but the core scheduler intentionally delegates those semantics to repository workflow instructions and agent tools. [sample workflow](https://github.com/openai/symphony/blob/7af5a7648c9fbffa08825fe0c0b18be00100aff3/elixir/WORKFLOW.md)
- **Inference:** Symphony materially strengthens the architecture decision and changes the implementation baseline: the first pilot should treat Symphony's spec as the closest upstream reference, adopt or adapt only the locally needed subset, and test the repository-specific authority/lease and durability gaps rather than independently inventing the entire controller. It does not authorize installing or running the preview, using Linear credentials, enabling `approval_policy: never`, unattended tracker/Git writes, or replacing independent acceptance gates.

### Durable graph/workflow frameworks

- **Verified:** LangGraph checkpoints graph state at each superstep and persists completed task writes so they need not rerun after a peer failure. `interrupt()` persists state and waits indefinitely until a caller resumes it; its documentation explicitly requires idempotent side effects before an interrupt. This is the closest agent-framework match to checkpoints + restart-safe progress + human gate, but it is not a lease or product authorization system. [Persistence](https://docs.langchain.com/oss/python/langgraph/persistence), [interrupts](https://docs.langchain.com/oss/python/langgraph/interrupts)
- **Verified:** Microsoft Agent Framework executes typed workflow graphs in BSP supersteps, validates graph construction, emits lifecycle/executor/superstep/request events, and can checkpoint full workflow state. It supplies in-memory, local-file, and Cosmos checkpoint storage; the Durable Extension additionally recovers progress across distributed workers. Approval-required tools and request ports pause the workflow and pending requests are restored with a checkpoint. This is an exact architectural precedent for deterministic graph gates, durable checkpoints, HITL, and bounded orchestration topology. [Workflow execution](https://learn.microsoft.com/en-us/agent-framework/workflows/workflows), [checkpoints](https://learn.microsoft.com/en-us/agent-framework/workflows/checkpoints), [HITL](https://learn.microsoft.com/en-us/agent-framework/workflows/human-in-the-loop), [Durable Extension](https://learn.microsoft.com/en-us/agent-framework/integrations/durable-extension)
- **Verified:** Google ADK defines sessions with chronological events/state, and
  ADK 2.0 now explicitly combines deterministic workflow logic with open-ended
  agent steps, human-in-the-loop nodes, and durable resume. Google argues that
  routing, scheduling, and error handling should remain deterministic rather
  than being delegated to an LLM. The sources still do not supply a repository
  authority record or lease. ADK Go 2.0 additionally documents per-node retry
  with backoff/jitter, graph-wide `WithMaxConcurrency(n)`, and idempotent resume;
  external side effects still require application policy. [ADK 2.0 rationale](https://developers.googleblog.com/why-we-built-adk-20/),
  [ADK Go 2.0 workflows](https://developers.googleblog.com/announcing-adk-go-20/),
  [ADK sessions](https://adk.dev/sessions/),
  [ADK human input](https://adk.dev/graphs/human-input/)
- **Verified:** Temporal guarantees recovery from crash/network/infrastructure failures through durable workflow execution. Its API exposes workflow-ID conflict policy, retry policy, and de-duplicated signal request IDs; its platform guidance identifies Signals/Updates as native human-approval gates. This is the strongest primary-source precedent for durable single-execution identity and idempotent retry, but it deliberately does not decide whether a particular business action is authorized. [Temporal documentation](https://docs.temporal.io/), [Temporal API protocol](https://api-docs.temporal.io/), [human-gate guidance](https://go.temporal.io/platform-hub/decision-framework)

### Coding harnesses and multi-agent products

- **Verified:** Claude Code subagents can have a tool allowlist/denylist, permission mode, `maxTurns`, skills, hooks, a worktree-isolation option, and persistent memory scope. A parent can restrict spawnable subagent types; subagents cannot spawn subagents. `PreToolUse` hooks and permission rules can block tool calls. These are direct precedents for bounded, non-recursive delegation and deterministic pre-action gates, not durable goal ownership or locking. Agent hooks are explicitly experimental. [Subagents](https://code.claude.com/docs/en/sub-agents), [hooks](https://code.claude.com/docs/en/hooks), [permissions](https://code.claude.com/docs/en/permissions)
- **Verified:** OpenHands documents an immutable, type-safe event system driving execution and state management, plus an agent state initialized from ordered events. This supports audit/trajectory-style observability. The reviewed official material did not establish durable checkpoint/replay, human approval, lease, or bounded multi-agent orchestration contracts. [OpenHands event system](https://docs.openhands.dev/sdk/arch/events), [Agent API](https://docs.openhands.dev/sdk/api-reference/openhands.sdk.agent)
- **Verified:** SWE-agent records a trajectory with turns, configuration, logs, environment state, patches, and exit statuses; it separates agent run from benchmark evaluation. Its `RetryAgent` re-queries after malformed output, blocked actions, or shell syntax errors. It is now maintenance-only and points users to mini-swe-agent, so it is useful as a coding-harness precedent but not a recommended new dependency. [Trajectories](https://github.com/SWE-agent/SWE-agent/blob/main/docs/usage/trajectories.md), [RetryAgent](https://swe-agent.com/latest/reference/agent/), [CLI status](https://swe-agent.com/latest/usage/cli/)
- **Verified with limited detail:** CrewAI documents Flows as stateful, persistent, resumable long-running orchestration and documents sequential/hierarchical/hybrid processes with guardrails, callbacks, human-in-the-loop triggers; its source docs expose `max_retry_limit`. Those claims establish feature direction, but the reviewed landing/source pages are less precise than LangGraph/Microsoft/Temporal about durability and concurrency semantics; do not treat them as a replacement for a lease or deterministic acceptance contract. [CrewAI documentation](https://docs.crewai.com/), [agent options source](https://github.com/crewAIInc/crewAI/blob/main/docs/edge/en/concepts/agents.mdx)

## Convergence and divergence

**Convergence (current trend):**

1. **Persistence is moving from chat history to explicit state.** LangGraph checkpoints, Microsoft checkpoints, Temporal histories, ADK session events, Agents SDK `RunState`, and SWE-agent trajectories all make resume/audit possible. The durability strength varies sharply: a transcript or trajectory aids reconstruction; a durable workflow engine replays a state machine.
2. **Human approval is a first-class pause/resume pattern.** OpenAI tool approval, LangGraph interrupts, Microsoft request/response, Google ADK human-input nodes, Claude permissions/hooks, and Temporal Signals/Updates all separate “wait” from “failure.” This is a current, cross-vendor pattern.
3. **Multi-agent delegation is increasingly constrained rather than free-form.** Handoffs/tool schemas, graph edges, typed messages, permission/tool allowlists, maximum turns, worktree isolation, and non-recursive child restrictions reflect a broad move toward bounded specialist work.
4. **Observability is normal, but it is not acceptance.** Traces, events, checkpoint histories, and trajectories are common. None turns model self-report into product-semantic completion; trusted commands, external evaluation, or an accountable human remain separate.
5. **Idempotency is an explicit reliability requirement at side-effect boundaries.** LangGraph calls this out as an application duty for interrupt replay; Temporal supplies durable retry/request IDs; ADK Go documents idempotent resume. Coding harnesses typically limit retries or preserve attempts but leave external side-effect idempotency to the application.
6. **Tracker-backed coding orchestration is now an explicit OpenAI pattern.** Symphony makes the issue tracker the work control plane, keeps workflow policy in the repository, and continuously reconciles isolated Codex runs. Its draft/prototype status and in-memory recovery limits show why the local pilot still needs evidence-gated hardening.

**Divergence (not a universal agent-harness feature):**

1. **A TTL lease / repository single writer is not standard in agent SDKs.** Temporal workflow IDs and CI concurrency groups are related coordination mechanisms, but none of the reviewed coding-agent runtimes supplies the proposed local worktree/HEAD-aware TTL lease. This is project-owned integration work, not a missing checkbox to solve by adopting an SDK.
2. **A durable *authority* record is stronger than a session, trace, or checkpoint.** Frameworks preserve execution state; they do not encode this repository’s approvals, product boundaries, privacy restrictions, allowed paths, or prohibition on signing/publishing/credentials. The project’s record is deliberately domain/policy-specific.
3. **Review-repair-validation is common as a pattern, not normally an automatic semantic gate.** Codex’s loop is a documented example; graph engines can model the stages. The exact no-progress fingerprint, independent verifier/validator split, and completion contract must remain local.
4. **Determinism is scoped.** Microsoft BSP scheduling and Temporal replay determinism constrain workflow mechanics. They do not make LLM outputs, browser state, Git worktrees, or product decisions deterministic. A deterministic local command gate still has separate value.

## Adoption recommendation

**Recommendation: retain the thin, local controller proposal, but use the Symphony specification as its primary upstream reference; do not install the preview or add a general orchestration framework now.** Symphony substantially reduces architectural uncertainty, but its own specification leaves durable authority, TTL leasing, restart-stable blocked state, end-to-end idempotency, and semantic acceptance to the implementation. A framework or wholesale preview adoption would add operational surface without removing that local policy work.

Adopt only these portable patterns if the root authorizes the one-goal pilot:

1. **Durable, minimal state:** one schema-validated active-goal record plus append-only evidence entries, treated like a local workflow history—not a transcript.
2. **Admission before execution:** acquire an atomic local one-writer lease, reconcile HEAD/worktree/owned paths, validate authority and acceptance inputs, then run one bounded unit.
3. **Wait as a terminal operational state:** use `waiting_for_human`/`blocked` for authority, ambiguity, approval, privacy, lease, or no-progress conditions; do not convert these into retries.
4. **Bounded quality loop:** only for a predeclared, local, deterministic failure class: read-only review → focused repair → independent deterministic validation → persisted fingerprint → either next bounded attempt or stop. Model judgment stays advisory.
5. **Bounded delegation:** preserve existing direct-child limits, exclusive paths/worktrees, named artifacts, and fresh independent verifier/validator gates. Do not recursively fan out.

Do **not** infer from any framework that the controller may add credentials, run unattended external writes, use a permissive approval mode, publish, sign, deploy, modify product/architecture scope, or process private relationship data. No reviewed platform supplies that authorization.

## Root decisions still required

1. Approve status quo versus a single local, disposable scheduled-goal pilot; this report supplies no scheduling authorization.
2. Choose the durable-record path/retention model and confirm it is the only operational authority alongside the append-only orchestration log.
3. Define lease identity, TTL, stale-takeover reconciliation, conflict behavior, and an explicit stop-on-lease-loss rule.
4. Approve the autonomous operation/path allowlist, maximum runs/turns/attempts, accepted deterministic commands, no-progress fingerprint, and notification channel.
5. Decide whether observed recurrence justifies a bounded repair-loop pilot; absent two comparable incidents, retain manual repair plus existing final gates.
6. Decide trace/checkpoint retention and redaction before collecting model or browser artifacts; exclude real URLs, calendars, addresses, private notes, relationship data, credentials, and raw browser state.

## Research gaps and refresh triggers

- No reviewed source establishes that any agent SDK can infer semantic ambiguity or repository authority safely. That remains a human-policy decision.
- This survey did not run provider code, create accounts, configure durable stores, or test scheduler behavior. Operational guarantees must be validated in an explicitly authorized local pilot.
- Refresh before adopting a framework, changing Codex/Claude/Agents SDK major behavior, enabling a scheduler/CI runner, or if a vendor introduces native repository locks, durable task authority, or formal semantic acceptance controls.
